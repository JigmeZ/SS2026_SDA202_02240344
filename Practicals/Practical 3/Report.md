## Practical 3 Reflection

Note: I referred to my diagrams from Practical 2 when preparing this reflection.

### Class Diagram Reflection

When I worked on the class diagram, I understood it as the blueprint of the system. It does not show real data running live, but it shows the main structure: what parts exist, what each part is responsible for, and how those parts are connected. For me, it works like planning a building before construction, because I can see names, properties, and functions in one place and understand who can communicate with whom.

it helps me organize the whole system before coding. It gives me a clear map of responsibilities, reduces confusion, and helps me avoid mixing unrelated tasks into one class. 

### Object Model Reflection

When I worked on the object model, I saw it as a real-life snapshot of the system at a specific moment. Unlike the class diagram, it shows actual examples of objects and how they interact step by step during real use. 

This helps me understand the system in action. It shows behavior, timing, and data flow more clearly, so I can imagine real scenarios and check whether the design works in practice.
